1. Who You Are

You are the Administrator. You hold the keys to the system: every menu, every record, every setting. You are also the first point of contact for every other user when they have an issue.

1.1 What you can do

  • Create, edit, disable, or delete user accounts.
  • Assign roles (initiator, HOD, procurement, finance, MD, admin).
  • Set the "Can Access Stores" flag for users who need the Stores module.
  • Manage the vendor master list.
  • Manage the client list.
  • Set department budgets and FX rates.
  • See every form in the system for support purposes.
  • Reassign stuck approvals when an approver is unavailable.
  • View the audit log for forensic / compliance requests.

1.2 What you should NOT do

  • Do not approve Purchase Requisitions or financial forms unless you hold a genuine secondary role — approval belongs to HODs, Finance, and MD.
  • Do not edit historical records to "fix" them. If a record is wrong, document the correction through proper channels.
  • Do not share your admin account credentials. If you need another admin, create them a separate account.

1.3 Your responsibilities

  1. Keep the user master list accurate (right role, right department, right HOD assignment).
  2. Respond to password resets within a few hours.
  3. Keep the vendor list clean and up to date.
  4. Set and review budgets and FX rates if Finance isn't able.
  5. Preserve the audit log — never manipulate it.
  6. Respond to support requests from all other users.
  7. Coordinate with IT on any backend issues (system down, database errors, etc.).

4. Task 1 — Create a User

4.1 Open the Users page

  1. Sidebar → Administration → Users.
  2. You see the full list of user accounts.

4.2 Click Add User

  1. Click the Add User button (usually top-right).
  2. A form opens.
Users page with Add User button at the top and a searchable list of existing users.

4.3 Fill in basic details

  1. Username — 3+ characters. Letters, numbers, dots, underscores, hyphens. Use a consistent convention (e.g. firstname.lastname).
  2. Password — minimum 8 characters with upper, lower, and digit. Use a strong one-time password that the user will change on first login.
  3. Full Name — as it should appear on approvals and PDFs.
  4. Email — the user's real work email address. Must be unique.
  5. Employee Number — HR identifier (optional but recommended).

4.4 Assign role and department

  1. Role — pick one:
    • initiator — any staff member who creates requests
    • hod — head of a department, first-stage approver
    • procurement — vendor management and adjudication
    • finance — second-stage approver, budgets, FX
    • finance_manager — same as finance with wider visibility
    • md — final approver on high-value PRs
    • admin — full system access (this role)
  2. Department — pick from dropdown.
  3. Is HOD — tick if this person is the head of their department (separate from the hod role).
  4. Supervisor Name — the full name of this user's HOD, used to route their requests.

4. Task 1 — continued

4.5 Set the Can Access Stores flag

  1. Tick Can Access Stores if the user should see the Stores section.
  2. This is independent of their role. A Finance officer who also handles GRNs needs this flag.

4.6 Save

  1. Click Save.
  2. The user account is created and active immediately.
  3. If email is configured, the user receives a welcome message.

4.7 Tell the user their password

  1. Tell the user their temporary password out-of-band — face-to-face, by phone, or via a secure channel. Not in plain email.
  2. Instruct them to change it on first login.
Never send passwords in plain email. If your email is intercepted, the attacker has both the username (visible in the From/To header) and the password. Use phone or in-person delivery.

4.8 Bulk user creation

  1. Use the Upload from Excel action (if available).
  2. Download the template, fill it in, upload.
  3. Review the import results — any errors appear row-by-row.

5. Task 2 — Reset a Password

5.1 Find the user

  1. Sidebar → Administration → Users.
  2. Search by name or username.
  3. Click the row to open their profile.

5.2 Reset action

  1. Click Reset Password.
  2. Generate a new temporary password that meets the requirements.
  3. Click Save.

5.3 Notify the user

  1. Tell them the new password face-to-face or by phone.
  2. Tell them to change it on first login.

5.4 Verify identity before resetting

  1. Confirm you're talking to the real user (voice recognition, face-to-face, or known email).
  2. Beware of social-engineering: an attacker pretending to be a user to get into an account.
  3. If the request feels wrong, ask for additional verification (employee number, HR date, etc.).
A password reset request that comes by email from an unknown sender is a red flag. Always verify the requester is genuinely the owner of the account.

6. Task 3 — Change a User's Role or Department

6.1 Open the user profile

  1. Sidebar → Administration → Users → click the user.

6.2 Change the relevant fields

  1. Role — use the dropdown.
  2. Department — use the dropdown.
  3. Is HOD — tick or untick.
  4. Supervisor Name — update if the reporting line changes.
  5. Can Access Stores — tick or untick.

6.3 Save

  1. Click Save.
  2. The change takes effect on the user's next login (or immediately if they refresh).

6.4 Ripple effects to consider

  1. If you demote someone from hod, any pending requests routed to them will need to be reassigned.
  2. If you change their department, historical requests stay tied to the old department (that's correct — spend was against the old dept's budget).
  3. If you change their supervisor, new requests will go to the new supervisor. Already-pending requests stay with the old one.

7. Task 4 — Disable / Deactivate a User

7.1 When to disable vs delete

  1. Disable when the user has left the company. They can no longer log in, but history and audit trail are preserved.
  2. Never delete a user who has ever initiated or approved anything. Deleting breaks the audit trail.

7.2 Disable a user

  1. Sidebar → Administration → Users → find and open the user.
  2. Change Status to disabled (or untick Active, depending on the UI).
  3. Click Save.
  4. The user can no longer log in. Their name and history remain visible in reports and audit.

7.3 Reassign their pending work

  1. Before disabling an HOD or approver, check their Pending Approvals queue.
  2. Reassign to another HOD (see Task 10) or explicitly rejection-with-reassignment.
An HOD who left with a full Pending Approvals queue can freeze the whole department's spending. Always reassign before disabling.

8. Task 5 — Manage Vendors

8.1 Open the Vendors page

  1. Sidebar → Administration → Vendors.

8.2 Add a vendor

  1. Click Add Vendor.
  2. Fill in: name, code, category, contact, email, phone, tier (1-3), rating (1-5), status.
  3. Save.

8.3 Edit a vendor

  1. Click a vendor row.
  2. Update the fields.
  3. Save.

8.4 Disable a vendor

  1. Change Status to inactive.
  2. Save.
  3. The vendor no longer appears in Procurement's Add-Quote dropdown but historical quotes remain intact.

8.5 Bulk upload from Excel

  1. Download the vendor template.
  2. Fill in columns: name, code, category, tier, rating, contact info, status.
  3. Click Upload from Excel.
  4. Review the import report.
Vendor codes should be stable. If you change a vendor's code, historical PRs using the old code still reference the old record. Prefer adding a new code and marking the old as inactive.

9. Task 6 — Manage Clients

Clients are external parties whose stock is reserved within your warehouse. Used on GRNs and Issue Slips when the Customer field is set.

9.1 Open the Clients page

  1. Sidebar → Administration → Clients.

9.2 Add a client

  1. Click Add Client.
  2. Fill in: name, code, contact, address, status.
  3. Save.

9.3 Edit / disable

  1. Same pattern as vendors: open, edit, save — or set status to inactive to disable.

10. Task 7 — Set Budgets

10.1 Open Budgets

  1. Sidebar → Financial Planning → Budgets.
  2. See every department with its annual budget and spend-to-date.

10.2 Set or update a budget

  1. Click the department row.
  2. Enter the annual budget in ZMW.
  3. Save.

10.3 Year-end rollover

  1. At financial year-end, reset each department's Spent to zero.
  2. Set the new year's budget values.
  3. Coordinate this timing with Finance.
  4. Document the rollover with a clear comment.

11. Task 8 — Set FX Rates

11.1 Open FX Rates

  1. Sidebar → Financial Planning → FX Rates.

11.2 Update a rate

  1. Click the currency.
  2. Enter the new ZMW rate (how many ZMW equal one unit of that currency).
  3. Save.
FX rates are normally set by Finance. Admin should only update them when Finance is unavailable. Document every change you make with the source (e.g. "Bank of Zambia mid-rate 21 Apr 2026").

12. Task 9 — Review the Audit Log

12.1 Why the audit log matters

  1. Every approval, rejection, and GRN status change is recorded permanently.
  2. It captures: actor name, IP, browser, from/to status, timestamp, comment.
  3. Used for forensic investigations, compliance, and fraud detection.

12.2 Open a record's history

  1. Open any form's detail page.
  2. Scroll to the Approval History section.
  3. See every action with who, when, and what.

12.3 Query the wider audit log

  1. A dedicated Audit Log page may exist under Administration.
  2. Filter by actor, entity type, date range, or action.
  3. Export to CSV for deeper analysis in Excel.
  4. If the UI does not expose this, IT can query the audit_logs MongoDB collection directly.
You cannot modify or delete audit log entries. If you think an entry is wrong, add an explanatory comment on the record itself — don't try to rewrite history.

13. Task 10 — Emergency Reassign a Stuck Approval

If an approver is unavailable (on leave, left the company) and their queue is blocking others, you can reassign.

13.1 Identify the stuck request

  1. Ask the initiator for the form ID (e.g. KSB-PR-20260421123456789).
  2. Find the form in the appropriate list.
  3. Check the current status and who is the current approver.

13.2 Decide the reassignment target

  1. If the original HOD is out: reassign to the deputy HOD, or a peer HOD of a related department.
  2. If Finance is out: reassign to another Finance officer or escalate to MD.
  3. If MD is out: coordinate with the Board for temporary delegation.

13.3 Perform the reassignment

  1. Methods depend on the UI:
    • If a reassign button exists: click it, pick the new approver, save.
    • If not: coordinate with IT to update the assigned_approver field via a maintenance script.
  2. Document the reassignment in the form's Comments with the reason (e.g. "Reassigned to John Banda because HOD Mary Mwale is on two-week leave.").

13.4 Notify the new approver

  1. Email the person and tell them to check their Pending Approvals.
  2. Tell the initiator too so they know their request is moving again.
Reassignments are visible in the audit log. Do them sparingly and only when necessary. Over-use undermines the approval workflow.

14. Troubleshooting

14.1 A user says "I can't log in"

  1. Verify their identity.
  2. Check their account status (active or disabled).
  3. Reset password (Task 2).
  4. If still failing, check they're using the right URL.
  5. If multiple users can't log in, suspect backend downtime — contact IT.

14.2 A user's menu is missing items

  1. Verify their role (Task 3).
  2. Verify the Can Access Stores flag if they need stores.
  3. Tell them to hard-refresh: Ctrl+Shift+R.

14.3 Stock register shows impossible numbers

  1. Rare but possible if a GRN was deleted outside the app or data was restored from backup.
  2. Coordinate with IT to reconcile. Do not attempt ad-hoc database edits — they will bypass the audit trail.

14.4 "FATAL: JWT_SECRET" error on login

  1. The backend server failed to start because the JWT secret is missing or too short.
  2. Coordinate with IT to check Render dashboard environment variables.
  3. Do not attempt to change secrets without IT involvement.

14.5 Email notifications are not going out

  1. Check your own inbox for test actions.
  2. Check the email service status (Resend, SMTP) with IT.
  3. Check user profiles have valid email addresses.

14.6 Forms are duplicating or creating with colliding IDs

  1. Extremely rare. Document the IDs.
  2. Contact IT — may indicate a timestamp-generation bug or a system clock issue.
